# 音乐层:三方协作设计 + 真选曲(方向 1+2)

- 日期:2026-06-26

- 背景文档:`docs/music-considerations.md`(现状盘点 + 缺口 + 四个方向)
- 本轮范围:把「选曲」从查表升级成按音频特征真选(方向 1+2),给方向 3 预留字段,不动剪辑结构

## 1. 背景与现状

现状链条见 `docs/music-considerations.md`。一句话:今天的音乐层 = 文字推断 1/4 情绪 → 取该情绪文件夹第一个文件 → 卡全局 tempo + 语音闪避 + 结尾淡出。几乎不考虑音乐本身。

代码事实(已核对):

- `composerv/music/library.py:32` `suggest_track` = `library[feeling][0]`,无任何特征。
- `composerv/music/mood.py:10` 情绪只有 4 个词,文字推断。
- `composerv/director/montage.py:64` 选曲发生在调用导演**之前**,导演只拿到 `track` 名 + `tempo` 数字。
- `composerv/director/prompt.py:61` 导演已输出一个 `arc` 字段,但是一两句散文,不是曲线。
- `composerv/models.py` 无任何音频特征模型,只有 `MusicBed` / `AudioHighlight`。

## 2. 目标与范围

### 2.1 要达到

- **真选曲**(方向 1+2):扩库 + 给每首曲子算音频特征;选曲从「取第一个」升级为「按这支成片要的情绪弧线 + 目标时长打分排序」;导演把「想要什么样的音乐」当成一个编辑决策输出。

### 2.2 本轮不做(但预留接口)

- **方向 3(音乐结构驱动剪辑)**:把最强镜头对到音乐高潮、切点卡乐句边界、挑曲子的合适片段对齐弧线。本轮在数据结构里预留 `phrase_boundaries`、`climax_t` 两个字段(先留空),不实现对应逻辑。

### 2.3 两个层面要分清(本设计的支点)

- **设计时**:用三个 subagent(配乐 / 导演 / 评估)按协议 C(propose-react-revise 协商)来回拉扯,把音乐层的提示词 + 打分函数调到三方满意。这是一种开发方法。
- **运行时(出货的成片管线)**:仍然**单 agent 串行**,不跑多 agent 协商。设计时协商产出的,是让这条单管线一次走对的提示词 + 打分函数。这与已记录的决定「多 agent 没验证前先单 agent 串行」一致。

## 3. 核心设计:共享弧线表示

让「又聋又瞎的导演」(只读文字)和「看不到画面的配乐」(只读音频特征)说上同一种话的关键,是把两边都归一化成同一条曲线,然后比形状。

- **时间归一化**:横轴压到 0(开头)到 1(结尾),重采样成固定 N 个点(本轮 N=16)。这样一支 90 秒的曲子和一段 30 秒的成片弧线,都变成「16 个 0 到 1 之间的数」,可直接比。
- **音乐侧曲线**:离线算每首曲子的能量(响度 / onset 强度随时间),归一化后存 16 个点。
- **画面侧曲线**:导演输出它**想要**的弧线,同样 16 个点。
- **匹配** = 比这两条 16 点曲线的形状接近程度,再加 tempo、大小调、valence、时长这几个标量。

一条真实测试成片(2025-12-14)的具体例子:导演想要的弧线 ≈ `0.2 安静引子 → 在 0.6 处升到 0.8(走到散步高潮)→ 结尾收到 0.3`;选曲就是找一首能量线形状最接近这个的曲子。

N=16 不是算出来的,是起步值,可调(见决策日志 D1)。

## 4. 数据模型

新增到 `composerv/models.py`(pydantic,与现有风格一致):

```python
class TrackFeatures(BaseModel):
    """一首曲子的离线音频特征,选曲时直接读。"""
    path: str
    duration_s: float
    tempo_bpm: float
    beat_times: list[float] = []          # detect_beats 已能出,复用
    mode: str = "unknown"                 # "major" | "minor" | "unknown"(近似)
    energy_curve: list[float] = []        # 16 点,0..1 归一化能量
    valence: float = 0.5                  # 0..1 明暗/正负向近似
    source: str = ""                      # 哪来的(库名 / URL),回溯用
    license: str = ""                     # 授权(CC0 / CC-BY / 用户自有 ...)
    # 预留方向 3(本轮留空):
    phrase_boundaries: list[float] = []   # 乐句边界(秒)
    climax_t: float | None = None         # 高潮时刻(秒)

class MusicIntent(BaseModel):
    """导演输出的『我想要什么样的音乐』,作为编辑决策的一部分。"""
    energy_curve: list[float] = []        # 16 点,导演想要的能量弧
    tempo_lo: float = 0.0                 # 想要的 tempo 区间(0 = 不限)
    tempo_hi: float = 0.0
    mode_pref: str = "any"                # "major" | "minor" | "any"
    valence: float = 0.5                  # 想要的明暗
    target_duration_s: float = 0.0
    arc_text: str = ""                    # 导演已有的一句话 arc,保留
```

## 5. 特征计算与存储

- **新增 `composerv/music/features.py`**:
  - `compute_features(audio_path) -> TrackFeatures`:用 librosa(已是依赖)算 tempo、beats、能量曲线(RMS / onset,重采样到 16 点并归一化)、mode(chroma + 调性模板,近似)、valence(大调 + 偏快 + 音色亮 → 高,近似)。
  - `read_sidecar(audio_path) -> TrackFeatures | None` / `write_sidecar(features)`:同名 `*.features.json`。
- **存储**:每首曲子旁放同名 sidecar JSON(`satie_gnossiennes.mp3` → `satie_gnossiennes.features.json`)。决策见 D2。
- **离线 index 命令**(新增,类似 `composerv music index <music_dir>`):扫库,给缺 sidecar 或源文件更新过的曲子算特征、写 sidecar。

### 5.1 音乐来源与授权

用户口味:游戏配乐风(尤其任天堂系列),涵盖 8bit / chiptune、电子、萨克斯、管弦。**任天堂原曲和 Spotify 上的二次创作录音都有版权,只当口味参照,不入库。**

曲库只收**真正免费 / 开放授权**、且对得上这个味道的文件。候选来源:

- OpenGameArt(CC0 / CC-BY,chiptune + 管弦游戏配乐)
- Incompetech(Kevin MacLeod,CC-BY)
- Free Music Archive、ccMixter、Pixabay Music(混合 CC / 免版税)
- 直接发 CC 授权的游戏音乐作者(如 Juhani Junkala 等)

每首入库时在 sidecar 记 `source` + `license`(决策 D7),授权始终可查,也对上「文档化、可回溯」的要求。

时机(用户定):这轮**不做大规模扩库**;实现阶段顺手下载一小批(覆盖 upbeat / calm / 8bit / 管弦几种),既扩库又当真选曲的测试素材(只有现在 4 首时,选曲是非题,没法验证)。大规模扩库另开。

## 6. 三个 subagent(设计时)

故意让每个 agent 只能看到它该看的,逼它们走共享曲线,以此验证「曲线表示够不够用」。每个 agent = 角色 + 能力约束 + 提示词 + 工作脚本(它能调的函数)。

### 6.1 导演 subagent
- 能力:只读文字(帧描述 + 字幕)。听不到、看不到。
- 产出:对一条测试成片输出 `MusicIntent`;并对配乐的候选做反应(挑一首,或说「高潮太早 / 能量太平,要 X」,即修订 intent)。
- 工作脚本:读该成片的 footage table → 输出 MusicIntent JSON;给定候选 → 输出「选 X」或修订后的 intent。

### 6.2 配乐 subagent
- 能力:只读音频特征(sidecar:曲线 / tempo / 大小调 / valence / beats)。看不到画面,读不到帧描述。
- 产出:拿导演的 MusicIntent,按打分函数(见第 9 节)排库,返回前 K 名候选,每首附音频实况 + 「为什么像 / 哪里不像」。被导演反驳就重排 / 换片段。
- 工作脚本:加载 sidecar 特征 → 算匹配分 → 返回排序候选 + 理由。

### 6.3 评估 subagent
- 能力:只读「计划表示」(谈定的意图曲线 + 选中曲子曲线 + 时间线计划 + 帧描述)。同样听不到看不到。
- 产出:判「过 / 不过 + 具体理由」:曲线对齐了没、切点相对节拍合不合理、选曲理由自洽不。
- 关键节点:由人(你)对真渲染出的成片做最终判断,评估 subagent 的自动判用于多数轮。

## 7. 设计时协商循环 + 收敛

每条测试成片跑一遍:

1. 导演 → MusicIntent
2. 配乐 → 前 K 候选 + 理由
3. 导演 → 挑一首 / 修订 intent(反应)
4. 配乐 → 重排(如需要)
5. 评估 → 对谈定计划出判词
6. 不过 → 带评估理由回到第 3 步;过 → 进关键节点

**收敛保护**:

- 回合上限(起步 4 轮 react,见 D5);超出则停,记录当前最优 + 原因。
- 平手裁决:没有候选过线时,选「最不差的」并置 `library_gap=True`,记「库缺这种曲子」。这是真实结论,不是崩溃。

**关键节点**:用产出的脚本在真实素材上真渲染,你看+听,认可或带批注退回。

**三方满意 = 评估过 + 你在关键节点认可。**

**循环真正产出的**(写进代码的东西):调好的提示词(导演怎么出意图、配乐候选怎么解释、评估的判据)+ 打分函数权重。

## 8. 出货运行流程(单 agent 串行)

1. 导演一次过,同时输出剪辑 + `MusicIntent`(在现有输出里加意图字段,它本就输出 `feeling` + 散文 `arc`)。
2. 非 LLM 打分函数(第 9 节)按曲线 + 标量从特征库选最佳曲子。
3. 卡拍后处理:把切点吸到选中曲子的真实 `beat_times` 上(用现有 detect_beats 的输出)。导演剪时还不知道最终曲子的精确节拍(曲子在它之后才定),所以由这步做对齐,导演只按目标 shot 长度剪。
4. `MontagePlan` 记下选了哪首 + 匹配分 + 两条曲线 + 卡拍调整 + 是否 library_gap。

运行时没有协商循环,一次 LLM 调用,符合「先单 agent」。

## 9. 选曲打分函数(起步版,协商时再调)

把意图曲线和每首曲子的能量曲线都取 16 点(都在 0..1)。总分 = 加权和:

- `shape`:两条 16 点曲线的形状接近度(起步用 1 − 平均绝对差,或归一化相关)。
- `tempo`:曲子 tempo 落在意图区间内得满分,越远越低(区间为 0 = 不计此项)。
- `mode`:大小调匹配(意图 `any` = 不计)。
- `valence`:`1 − |曲子.valence − 意图.valence|`。
- `duration`:曲子(或其一段)能否覆盖目标时长。

起步权重 `shape 0.5 / tempo 0.2 / valence 0.15 / mode 0.1 / duration 0.05`(见 D4,含证伪条件)。权重是协商要调的旋钮,最终值写进决策日志。

## 12. 错误处理

- 无特征 / 库为空 → 退回老 `suggest_track`。
- 无候选过线 → 选最不差 + `library_gap=True` + 记日志,不崩。
- 渲染失败(关键节点)→ 报出来,不让协商循环崩。
- 导演意图残缺 / 解析失败 → 退回 4 桶情绪选曲。

## 14. 决策日志

每条:决定 / 理由 / 否决的替代 / 证伪条件(怎样算这个决定错了)。

- **D1 曲线点数 N=16**
  - 理由:够描「安静 → 推 → 顶 → 收」的大走势,又不至于细到把无意义抖动算进去。
  - 否决:N=4(太糙,分不出高潮位置)、N=64(细到拾噪声、匹配不稳)。
  - 证伪:若不同曲子的 16 点曲线区分度不够(都长得差不多、分不开),或匹配结果对 N 在 8/16/32 之间高度敏感,则 N 选错,需重定。
  - 构建结果(2026-06-26):N=16 保留。起步曲库 7 首的 16 点曲线已确认互不雷同(可区分,见 build log)。尚未经真协商在真实成片上校验(协商未跑,见未决项)。
- **D2 特征存 sidecar(非单清单)**
  - 理由:特征跟文件待在一起;加一首歌 = 丢文件 + 重算这一首;JSON 可直接看;算错删掉重生成。
  - 否决:全库单 `index.json`(加歌 / 并发要回写整文件)、塞进现有 store(库是用户自有文件,不该进 DB)。
  - 证伪:若 sidecar 与源文件频繁不同步导致读到陈旧特征,且无法用「源文件 mtime 比对」可靠检测,则该存法有问题。
- **D3 valence / mode 用近似估法**
  - 理由:librosa 能给可用的近似,本地优先、不引外部服务;选曲是排序问题,不需要乐理精确。
  - 否决:接外部音乐分析服务(违背本地优先)、人工标注(库要长大,手标不可持续)。
  - 证伪:若近似 valence / mode 与人耳判断在测试曲上经常相反(如把明亮曲判成暗),则估法不够用,需换方法或降权。
- **D4 打分起步权重 shape 0.5 / tempo 0.2 / valence 0.15 / mode 0.1 / duration 0.05**
  - 理由:形状(情绪弧)最重要,故占一半;tempo 影响卡拍体感次之;mode / valence 调色;duration 多为可裁,权重最低。
  - 否决:等权(忽略形状的主导地位)、只看 tempo(回到旧的卡拍思路)。
  - 证伪:若协商中评估 / 人反复推翻打分头名,且换权重能稳定改善,则起步权重错,以协商收敛后的值为准并更新本条。
  - 构建结果(2026-06-26,初版):起步权重保持不变。tempo 未约束轴从返回 0.0 修为 1.0,与 `_mode_score('any')` 一致,用户确认。设计时协商尚未跑真实成片。
  - 构建结果(2026-06-26,测试成片协商后更新):权重已修订为 `shape 0.6 / tempo 0.15 / valence 0.12 / mode 0.08 / duration 0.05`(总和仍为 1.0)。同步加入 `SHAPE_FLOOR = 0.85` 软门槛:shape < 0.85 的曲子按 `(shape/SHAPE_FLOOR)^2` 折算总分,无论其他轴多高都无法靠拢。修订原因:评估 subagent 在该次协商中三次推翻 0.877 综合分(shape 仅 0.795)的头名选曲,认为能量曲线偏离是视频可感知的失配;逻辑为「其他轴(tempo/mode/valence/duration)满分容易,shape 一旦弱就无法靠这些轴补回」。本次修订取代原 D4 起步值。
- **D5 协商回合上限 4**
  - 理由:防来回不收敛;4 轮足够让导演表达一次反驳 + 配乐改一次 + 复核。
  - 否决:无上限(可能死循环 / 烧 token)、上限 1(等于没有协商,退化成协议 A)。
  - 证伪:若多数测试成片在 4 轮内无法达到评估过线,说明上限定低或表示能力不足,需上调或回头改表示。
- **D6 出货管线维持单 agent 串行,协商只在设计时**
  - 理由:与已记录决定「多 agent 没验证前先单 agent」一致;运行时一次 LLM 调用,稳定可控。
  - 否决:运行时跑多 agent 协商(本轮不验证多 agent,且会推翻既有决定)。
  - 证伪:若设计时调出的单管线提示词 + 打分,在新成片上选曲质量明显不如设计时协商出的结果,说明「静态脚本」不足以替代运行时协商,需重新评估是否把协商搬到运行时。
- **D7 曲库只收免费 / 开放授权文件;任天堂 / Spotify 仅作口味参照;每首记 source + license**
  - 理由:与已定「曲子是本地文件、用户拥有」一致;开放授权文件本地化后授权清楚、可分享;Spotify 不能合法 / 技术上当文件源。授权可查也对上用户「文档化、可回溯」的要求。
  - 否决:直接用任天堂原曲 / Spotify 二创(有版权,无法合法入库 / 分享)、不记授权(库长大后无法回溯)。
  - 证伪:若开放授权库里找不到足够覆盖用户口味(8bit / 电子 / 萨克斯 / 管弦)的曲子,导致真选曲缺料,则「只靠免费源」不够,需重新考虑(如引导用户放自有文件)。
- **D8 自己生成音乐(方向 4)本轮不做,只记候选开源实现 + 约束**
  - 理由:方向 4 是最大工程,与本轮「真选曲」目标解耦;用户明确这轮不做。先验证选曲机制再谈生成。
  - 否决:这轮就接生成(范围膨胀、风险高)、从头写生成(用户明确反对)。
  - 证伪:若现有 + 可下载的免费曲库长期满足不了成片需求(总是没有合适的曲子),说明「选 + 收」不够,生成的优先级要提前。

## 15. 范围外 / 预留(方向 3 + 方向 4)

方向 3(音乐结构驱动剪辑):

- 不实现:乐句 / 高潮 / 段落分析,最强镜头对高潮,切点卡乐句边界,挑曲子合适片段对齐弧线,音乐反过来改剪辑结构。
- 已预留:`TrackFeatures.phrase_boundaries`、`TrackFeatures.climax_t`(本轮留空)。

方向 4(自己生成音乐):

- 本轮不做(用户定),记为未来方向,真做时单开一轮 brainstorming。
- 约束:用现成开源实现,不从头写生成。候选:Meta AudioCraft / MusicGen(开放权重,按风格提示生成音频);8bit 可走符号 / MIDI + chiptune 合成器。
- 决策见 D8。

## 16. 风险与未决项

- 画面侧曲线由导演「凭文字想象」给出,可能与真实观感有差。靠关键节点的真渲染 + 你拍板纠偏。
- valence / mode 近似的准确度未知,实现时拿真实库曲核对后再决定是否保留 / 降权(见 D3 证伪)。
- 测试素材目前确定的只有一条,需再找 1 到 2 条不同情绪成片;若找不到,设计时调参会偏向单一案例(风险,要在构建日志里标注)。
