# 本地理解层:加速 + 预处理 重构计划(2026-06-20)

## 实测依据(profiling 32B,34s 片)
| 变体 | 帧数 | 分辨率 | prefill token | 总耗时 |
|---|---|---|---|---|
| A | 34 | 720p | 40,788 | 515s |
| B | 8 | 720p | 9,639 | 95s |
| C | 8 | 512 | 1,511 | 26s |
- 加载 3.3s(mmap,基本不算成本 → 常驻模型只是小优化)。
- A 拆分:prefill 422s(占 82%)+ decode 92s。**成本几乎全在图像 prefill**,prefill ∝ 帧数 × 每帧分辨率(token 数)。
- 少帧 34→8:**5.4x↓**(token 近似线性)。降分辨率 720p→512:**再 3.7x↓**(每帧 token 1205→189,6.4x)。两者叠加 34满→8小:**~20x↓,26s,比实时快**。
- 取舍:8 帧已经不是「秒级」(34s 片只 0.23fps);512 可能丢人脸/文字细节。所以要靠「内容感知抽帧」在少帧下保住关键时刻,靠「可调分辨率」保住该看清的东西。速度不是问题,精度才是要调的。

## 优先级(按实测排序)
- **P1 内容感知抽帧(少帧)**:按镜头/动作变化抽,34 → ~8 帧。实测 5.4x↓。重点是少帧还能保住关键时刻(靠场景/运动选帧,而非均匀)。
- **P2 分辨率旋钮(每帧 token 少)**:720p → 长边 512 等,经 Qwen2.5-VL 的 min/max_pixels。实测 3.7x↓(每帧 token 6.4x↓)。要在精度需求下选档(人脸/文字可能需要更高)。
- **P3 预处理层**:产出 P1 要用的场景/选帧,并附带人脸/质量/音频信号。
- **P4 常驻模型**:只省约 3s/片,低优先,但顺手做(用 Python API,不再每片起子进程)。
- **P5 修时间戳对齐 bug**:模型返回条数 ≠ 帧数时,按帧强制对齐(截断/补齐),不要整体塌成 0.0。
- 贯穿:所有质量参数做成可调选项;量化档(4bit/8bit)做成可实验。

## 预处理层(都便宜、无需 VLM,可与 GPU 重叠)
1. 镜头/场景边界(ffmpeg `select='gt(scene,…)'` 或 PySceneDetect)→ 把片切成场景。
2. 内容感知选帧:在场景边界 + 运动峰值处选帧 → 少而准。
3. 运动/salience:逐秒帧差或光流幅度 → 哪几秒在发生事。
4. 画质:模糊(Laplacian 方差)、曝光、抖动 → 标出废片。
5. 人脸:insightface 检测 + 向量 → 人物(对应人脸标记需求)。
6. 音频:抽轨 + VAD(人声活动)+ 响度 → 节奏 + 哪些片有对白。
7. 近重复:感知哈希(pHash)→ 雷同片段归组。
8.(可选)帧向量(CLIP 风格)→ 语义检索 / 去重。

## VLM 主步(贵的那步)
- 吃预处理选出的少量关键帧,按设定分辨率喂常驻 32B。
- 每条描述对齐到真实帧时间戳(见 P5)。

## 分档使用
- 全库:预处理 + 便宜模型/粗理解 → 总览、挑片。
- 选中片段:常驻 32B 做秒级 → 精确切点。
- 云端 Claude:暂缓,先把本地跑通。

## 优化维度全集(供后续取用)
- A 时间(抽多少帧、抽在哪):帧率;内容感知抽帧;用视频自带 I 帧当采样点;两遍(粗扫定位 + 只在有意思处密采)。
- B 空间(每帧多贵):分辨率 / 像素预算;裁剪到感兴趣区域。
- C 模型 / 计算:模型大小 + 量化;常驻加载;提示缓存(固定系统提示的 prefill 复用);投机解码;一个实例内批处理。
- D 输出:max_tokens;每帧短语而非整句;greedy(已用)。
- E 流水线 / 调度:CPU 抽帧与 GPU 推理重叠;分档(便宜模型/云端做粗、32B 做精);结果缓存。

## 编码:帧间 vs 帧内压缩
- 不影响模型成本:模型看的是解码后的 RGB,不是压缩码流。模型成本只取决于 帧数 × 分辨率 × 模型。
- 影响「抽帧 / seek」:长 GOP 的帧间压缩,要解码任意时间点得从前一个 I 帧解起,跳着抽很多帧就慢;短 GOP / 全 I 帧每帧独立可解,seek 快(文件大)。
- 策略:代理用短 GOP 或全 I;或直接用源里的 I 帧当采样点(I 帧常落在镜头切换处,既快又有意义)。硬件解码(VideoToolbox)也加速抽帧。
- 结论:编码加速的是「抽帧」(相对推理是小头);它帮到大头的唯一方式,是用 I 帧 / 镜头结构去挑更少更好的帧 → 回到 A 类内容感知抽帧。

## 多模型并行(128GB 内存)— 测前判断
- 内存够:32B-4bit 约 18GB,128GB 放得下 4-5 份。但**瓶颈是单块 GPU,不是内存**。
- N 份模型共享同一个 GPU,只是分时轮流跑,总吞吐约等于不变,还多了切换/竞争开销。**得不到 N 倍加速。**
- 单 GPU 上正确的并行:(1) 在一个实例内对多段做 batch(若 mlx-vlm 的多图 batch 支持到位);(2) CPU 抽帧与 GPU 推理重叠成流水。
- 真正的吞吐提升来自实测的「减帧 + 降分辨率」(直接砍 GPU 工作量),不是多开模型。
- 可做的小实验(预期收益小):2 个 32B 并发 vs 1 个,看 decode 的内存带宽空隙能否被另一个的 prefill 填上 —— Apple Silicon 上通常很微弱。
